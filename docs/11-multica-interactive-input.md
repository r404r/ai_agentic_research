# multica 中「交互式 skill 输入」的处理方式

> 范围：本文回答一个具体的运行时行为问题——在本地直接跑 Codex CLI / Claude Code 时，skill 常常**当场暂停、要求用户做选择或追加输入**；这类交互在 multica 中如何流转，用户该在哪里补充信息。
>
> 性质标注：本文结论来自 multica 的运行时交互模型（一次性、非交互的评论驱动 run）。涉及具体源码路径的部分见 [证据索引](90-evidence-index.md) 与 [docs/10-multica.md](10-multica.md)；带有「设计取向 / 建议」的判断属于综合，已在文中标注。

## 核心结论：multica 里没有「实时交互」，所有交互都被转成 issue 评论的异步往返

在本地直接跑 Codex CLI 或 Claude Code 时，skill 经常会**当场暂停、弹一个选择/输入提示**，用户在终端里敲回车就继续了——这依赖一个活着的交互式 TTY（stdin）。

multica 的运行时模型不一样：**每次触发都是一次性的、非交互的批处理 run**。agent 被 issue 指派或新评论唤起 → 在一个隔离的 workspace 里跑到结束 → 退出。这一次 run 期间**没有可以挂起等待用户输入的通道**，stdin 也不接键盘。所以「skill 想问一句再继续」这件事，不能靠阻塞等待，而是被改写成下面这套流程。

## 一、需要用户做选择 / 补输入的场景，实际怎么流转

1. **agent 跑到需要用户决策的那一步时，不会卡住等输入**。如果硬等 stdin，结果只会是空输入或挂到超时，得不到答案。
2. **agent 把「问题」变成一条 issue 评论**：列出待确认项、可选方案、以及它打算采用的默认假设，然后**结束这一轮 run**。
3. **用户在该评论的 thread 里回复**，把选择 / 补充信息写进去。
4. **该回复本身就是一个新的触发器**，会给 agent 重新排一轮 run。新的这轮里 agent 能读到整个 thread（包含刚补的内容），接着往下做。

一句话：本地的「同步弹窗 → 敲键盘」在 multica 里变成了「**评论提问 → 回评论 → 下一轮 run 接力**」的异步往返。一次「交互」= 一个评论回合。

## 二、用户该在哪里补充信息

- **首选：直接在 issue 的评论 thread 里回复** agent 那条提问评论（保持同一 thread）。这条回复会自动重新唤起 agent，它会读取 thread 上下文继续。
- **更省事：在最初的 issue 描述 / 第一条请求里就把选择和参数前置写清楚**。例如「用方案 A」「目标分支是 xxx」「不要动数据库」。把已知决策一次性给足，agent 就能基于明确假设直接推进，省掉一轮往返。
- 用户**不需要、也无法**在终端里给输入——workspace 是平台托管的，用户看不到它的 TTY，唯一对用户可见的双向通道就是 issue 评论。

## 三、两类「交互」的处理差别（便于预期行为）

- **流程型 skill 的来回**（如 brainstorming、需求澄清这类，本质是 agent 主动想确认需求）：agent 会把澄清问题**攒成一批写成评论**，而不是一问一答地零碎打断用户。所以通常一次能看到它所有想确认的点，回一次即可。
- **技术上依赖 TTY 的交互 skill**（脚本里真的去 read stdin、弹 y/N 确认）：这类在非交互 run 里**没有活的输入通道**，agent 的正确做法是不阻塞——要么按安全默认值推进并在评论里说明，要么把决策点上报成评论让用户定夺。如果某个 skill 表现得像在「干等输入」，那其实是它没被正确改写成评论往返，属于应被反馈修复的问题。

## 实操建议（综合判断）

- 想减少往返：**在 issue 描述里前置决策**（方案、分支、范围、禁止项）。
- 中途要补：**回 agent 提问那条评论**即可，它会自动接力。
- 心智模型：把 multica 的 agent 当成「**邮件 / 工单式协作者**」而不是「终端里的实时对话」——它每次只在被触发时醒来干一轮，交互的颗粒度是「一个评论回合」，不是「一次按键」。
